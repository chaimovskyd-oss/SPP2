SPP2 — תוכנית חדשה למנוע פריסטים, תיקוני תמונה ואפקטי עמוד

אפיון מעודכן ומלא להחלפת מימוש שכבות ההתאמה הקלאסיות הנוכחי, שמכביד על התוכנה וגורם להיעלמות תמונות, במערכת יציבה יותר: תיקוני תמונה לא־הרסניים, פריסטים חכמים, אפקטי עמוד עליונים, ו־ספריית כלים מסודרת.

החלטה מרכזית

בשלב הראשון לא מיישמים שכבות התאמה קלאסיות בסגנון פוטושופ, כלומר שכבה שיושבת באמצע הערימה ומשפיעה על כל מה שמתחתיה. במקום זה בונים מנוע שמתאים יותר ל־SPP2:

Photo / Print Presets → נכתבים כתיקונים לא־הרסניים על ImageLayer Page Looks → שכבת אפקט עליונה וקלה על כל העמוד Smart Fixes → מתכון חכם שמפעיל תיקוני תמונה / עמוד לפי הצורך Classic Adjustment → נשאר Disabled / Experimental להמשך בלבד

כך שומרים את כל החזון של ספריית פריסטים, Looks, Photo Rescue וכלי דפוס — בלי לשבור את הקנבס.

1. למה משנים כיוון

הבעיה במימוש הנוכחי

שכבות ההתאמה הנוכחיות מנסות לעבוד כמו פוטושופ: שכבה באמצע הערימה משפיעה על כל השכבות שמתחתיה. בפועל, בתוך React-Konva זה גורם לעטיפת שכבות בקבוצה, cache גדול, חישוב כבד, ובחלק מהמקרים תמונות נעלמות.

הבעיה היא לא הרעיון של תיקונים ופריסטים. הבעיה היא הדרך שבה מנסים לרנדר אותם בזמן אמת.

מה מסוכן כרגע

מנגנוןלמה הוא בעייתימה עושים במקום
Adjustment Layer באמצע הערימה דורש לדעת מי מעל, מי מתחת, מי בתוך קבוצה, מי כבר נטען, ומה צריך cache. מכבים בשלב ראשון. משאירים כ־Experimental.
Konva Group cache לכל העמוד כבד מאוד, במיוחד בעמודים גדולים והרבה תמונות. יכול לתפוס cache לפני שהתמונה נטענה. לא עושים cache מלא לעמוד. תיקונים מוחלים על תמונה בודדת או אפקט קל עליון.
Preset כקבוצת שכבות רינדור אמיתיות יפה בממשק, אבל מסוכן אם כל שכבה דורשת קומפוזיציה. Preset Group הוא קבוצת הגדרות / מתכון, לא בהכרח קבוצת Konva כבדה.
קבוצות שכבות עם רינדור כבד מכניס סיכון ל־visibility, opacity, order, parentId, childIds. קבוצות הן metadata/UI בשלב ראשון.

מה כן רוצים לשמור

  • ספריית כלים עשירה ומסודרת במקום תפריט + ענק.
  • כפתורי קסם כמו Sun Rescue, Dark Photo Fix, Gold Noir, Sublimation Boost.
  • אפשרות לערוך את הפריסט אחרי ההפעלה.
  • אפשרות להחיל על תמונה אחת, כמה תמונות, כל התמונות בעמוד, או כאפקט עמוד.
  • Undo/Redo יציב.
  • ייצוא/הדפסה באיכות מלאה.

2. המודל החדש — לא שכבות התאמה קלאסיות, אלא SPP Smart Presets

1. Image Adjustments

תיקונים לא־הרסניים שנשמרים ישירות על ImageLayer.

מיועד לתיקון אמיתי של תמונות: בהירות, קונטרסט, חשיפה, רוויה, חום/קור, שחור לבן, חדות.

מהיריציבלא הורס מקור

2. Smart Presets

מתכונים שמחילים סט תיקונים על תמונה אחת, בחירה מרובה, או כל התמונות בעמוד.

לדוגמה: Sun Rescue מחיל Highlights/Shadows/Exposure/Vibrance על התמונות שנבחרו.

כפתור אחדניתן לעריכהUndo אחד

3. Page Look

אפקט עמוד עליון, קל, תמיד מעל הכול. מיועד לאווירה כללית ולא לתיקון תמונה מדויק.

לדוגמה: Sepia Wash, Warm Glow, Vignette, Color Overlay, Vintage Atmosphere.

תמיד למעלהקל לרינדור

4. Experimental Classic Adjustment

שכבת התאמה אמיתית שמשפיעה על כל מה שמתחת — לא נמחקת, אבל כבויה ולא בשימוש בשלב הראשון.

תחזור רק בעתיד עם מנוע קומפוזיציה אמיתי.

לא לשלב ראשוןעתידי

3. החלפת שכבות ההתאמה הקיימות בשיטה החדשה

מטרת ההחלפה

לא מוחקים את המידע הקיים ולא שוברים פרויקטים ישנים. בשלב הראשון מנטרלים את נתיב הרינדור המסוכן, וממפים את השימושים המרכזיים למנוע החדש.

חוקי Safe Mode

const ENABLE_CLASSIC_ADJUSTMENT_LAYER_RENDERING = false;
const ENABLE_FULL_PAGE_ADJUSTMENT_CACHE = false;
const ENABLE_PAGE_LOOK_OVERLAYS = true;
const ENABLE_IMAGE_LEVEL_ADJUSTMENTS = true;
שכבה קיימתמה עושים בטעינהמה המשתמש רואה
brightness/contrast adjustment layer אם היא גלובלית/מתחת — מסמנים כ־Classic Disabled. אפשר להציע Convert to Page Look או Apply to Images. שכבה קיימת עם תווית “מצב ישן / לא פעיל ברינדור חי”.
sepia / tint layer אפשר להמיר ל־Page Look אם היא אמורה להשפיע על כל העמוד. אפקט עמוד עליון עם strength.
preset group ישן להמיר ל־Smart Preset Instance אם אפשר. מתכון עריך, לא קבוצת Konva כבדה.
group layer לנרמל parentId/childIds ולהשאיר כ־UI metadata. קבוצה בלוח שכבות, בלי cache/transform כבד.

חשוב

לא לבצע המרה אוטומטית אגרסיבית שעלולה לשנות עיצוב ישן. בפרויקטים קיימים עדיף לטעון בבטחה, להציג אזהרה קטנה, ולאפשר למשתמש להמיר ידנית.

12. פרומפט זהב לקלוד / Cursor

להעתקה כמשימת פיתוח מלאה:

We need to replace the current unstable Photoshop-style Adjustment Layer implementation in SPP2 with a safer SPP-style Smart Preset architecture.

Context:
SPP2 is a React + Electron + React-Konva editor for print preparation, photo layouts, collage, product design, and print workflows.
The current adjustment layer implementation is causing performance issues and disappearing images.
The likely cause is that classic adjustment layers can sit in the middle of the layer stack and affect all layers below them, using cached Konva groups / full-page cache.
This is too heavy and unstable.

Main decision:
Do NOT continue implementing Photoshop-style adjustment layers as the main engine.
Do NOT use full-page Konva Group cache for adjustment layers.
Do NOT wrap all lower layers in cached groups.
Do NOT let adjustment layers render as “all layers below” in the live canvas for now.

Instead, implement a new architecture with three safe concepts:

1. Image Adjustments
2. Smart Presets
3. Page Look Layers

Classic Photoshop-like adjustment layers must remain disabled/experimental for now.

====================================================================
PART 1 — Disable the dangerous current path
====================================================================

Add feature flags:

const ENABLE_CLASSIC_ADJUSTMENT_LAYER_RENDERING = false;
const ENABLE_FULL_PAGE_ADJUSTMENT_CACHE = false;
const ENABLE_IMAGE_LEVEL_ADJUSTMENTS = true;
const ENABLE_PAGE_LOOK_LAYERS = true;

When ENABLE_CLASSIC_ADJUSTMENT_LAYER_RENDERING is false:
- Do not render classic adjustment layers using cached Konva groups.
- Do not call node.cache() over the full page for adjustment layers.
- Do not wrap layers below adjustment layers in a special render group.
- Existing adjustment layers may remain in the layer list as Legacy / Experimental / Disabled.
- Existing projects must open safely and images must not disappear.
- Optionally show a small warning in the UI: “Classic adjustment layer rendering is disabled for stability. Convert to Image Preset or Page Look.”

Do not delete user data automatically.
Do not convert old adjustment layers automatically unless the user explicitly chooses a conversion.

====================================================================
PART 2 — Implement Image Adjustments
====================================================================

Add non-destructive image adjustments directly to ImageLayer.

Each ImageLayer should support:

imageAdjustments?: {
  enabled: boolean;
  stack: ImageAdjustment[];
  presetInstances?: AppliedPresetInstance[];
}

Supported adjustment types for phase 1:
- basicTone:
  brightness, contrast, exposure, gamma, offset
- color:
  saturation, vibrance, temperature, tint, hue
- blackWhite:
  strength and optional channel mix values
- sepia/colorize:
  hue, warmth, strength
- detail:
  sharpness, clarity, noiseReduction
- curves:
  preset-based first, pro points later
- highlightsShadows:
  highlights, shadows, whites, blacks

Important:
- The original image asset must never be modified.
- Adjustments are stored as parameters only.
- Reset removes/disables the adjustment parameters and restores the original appearance.
- Rendering should apply filters only to the specific image node, not the whole page.
- If a filter fails, render the original image rather than hiding it.
- Avoid heavy synchronous processing during drag/zoom/page switch.

Add store actions:
- updateImageLayerAdjustments(layerId, patch)
- setImageAdjustmentStack(layerId, stack)
- addImageAdjustment(layerId, adjustment)
- removeImageAdjustment(layerId, adjustmentId)
- toggleImageAdjustment(layerId, adjustmentId)
- resetImageAdjustments(layerId)
- copyImageAdjustments(sourceLayerId)
- pasteImageAdjustments(targetLayerIds)
- applyAdjustmentToSelectedImages(adjustment)
- applyAdjustmentToAllImagesOnPage(pageId, adjustment)

Undo/redo:
- Applying a preset to multiple images should be one undoable action if the current undo system allows it.

====================================================================
PART 3 — Implement Smart Preset Engine
====================================================================

Create a preset registry with structured preset definitions.

Preset definition shape:

type SmartPresetDefinition = {
  id: string;
  name: string;
  category:
    | "Basic"
    | "Photo Rescue"
    | "Portrait"
    | "Print"
    | "Looks"
    | "Creative"
    | "Advanced"
    | "Experimental";

  description: string;
  tags: string[];
  defaultStrength: number;

  recommendedApplyMode:
    | "singleImage"
    | "selectedImages"
    | "allImagesOnPage"
    | "pageLook";

  allowedApplyModes: Array<
    | "singleImage"
    | "selectedImages"
    | "allImagesOnPage"
    | "pageLook"
  >;

  requires: string[];
  optionalRequires: string[];

  imageAdjustments?: ImageAdjustmentDefinition[];
  pageLookEffect?: PageLookEffectDefinition;

  notRecommendedAsPageLook?: boolean;
  notRecommendedForText?: boolean;
  printWarnings?: string[];
};

Preset application rules:
- Photo Rescue presets should usually apply to selected images or all images on the page.
- Print presets should usually apply to selected images or all images on the page.
- Looks can apply either as image presets or as Page Look layers depending on the preset.
- Page Look presets create lightweight top-level PageLookLayer objects.
- Do not create Photoshop-style “adjustment layer below stack” objects from presets in phase 1.
- Do not create Konva cached groups for presets.

When applying a preset to images:
- Generate one or more ImageAdjustment entries and append them to each target ImageLayer adjustment stack.
- Store an AppliedPresetInstance on the image so the preset can be edited/reset later.
- Apply master strength by scaling the generated adjustment values or storing a strength multiplier.
- Allow editing the generated adjustment values later.

AppliedPresetInstance shape:

type AppliedPresetInstance = {
  id: string;
  presetId: string;
  name: string;
  appliedAt: number;
  strength: number;
  targetMode: "singleImage" | "selectedImages" | "allImagesOnPage";
  editable: boolean;
  generatedAdjustments: string[];
};

====================================================================
PART 4 — Implement Page Look Layers
====================================================================

Add a new lightweight layer type:

type PageLookLayer = {
  id: string;
  type: "pageLook";
  name: string;
  enabled: boolean;
  locked?: boolean;
  opacity: number;
  strength: number;
  alwaysTop: true;
  effect:
    | { type: "colorOverlay"; color: string; blendMode: "normal" | "multiply" | "screen" | "overlay" | "soft-light"; opacity: number }
    | { type: "gradientOverlay"; stops: GradientStop[]; angle: number; blendMode: string; opacity: number }
    | { type: "vignette"; amount: number; softness: number; centerX: number; centerY: number }
    | { type: "grain"; amount: number; size: number }
    | { type: "pageWash"; warm: number; cool: number; sepia: number; fade: number; vignette?: number; grain?: number };
};

Rules:
- PageLookLayer always renders above all normal layers.
- PageLookLayer must never be placed inside a group.
- PageLookLayer must not be affected by group visibility/opacity.
- PageLookLayer is lightweight and should use simple Konva shapes, gradients, opacity, and blend modes when possible.
- Do not use full-page pixel filters for PageLook in phase 1.
- PageLook is for atmosphere, not precise photo correction.
- Examples: warm tint, cool tint, sepia wash, lighten wash, darken wash, vignette, gradient overlay, soft glow, grain.

====================================================================
PART 5 — Layer groups stabilization
====================================================================

Groups should be metadata/UI only in this phase.
They should not become heavy Konva render groups.

Supported now:
- Organize layers in the layer panel.
- Group visibility affects child visibility.
- Group lock affects child editability.
- Optional group opacity can multiply child opacity without cache.
- Delete group only: ungroup children.
- Delete group with children: delete children.
- Duplicate group: duplicate children safely.

Not supported now:
- group transform
- group clipping
- group masks
- nested groups
- group-level adjustment rendering
- adjustment layers inside groups
- pageLook layers inside groups
- group cache

Add normalizeLayerGroups(layers):
- Remove parentId if the parent group does not exist.
- Rebuild every group.childIds from actual layer.parentId values.
- Remove stale childIds.
- Remove duplicate childIds.
- Prevent nested groups for now.
- Prevent pageLook and legacy adjustment layers from being children of normal groups.
- Ensure pageLook layers are always topmost.
- Do not allow a broken group relationship to hide unrelated image layers.

Run normalization after:
- loading a document
- moving layers into/out of groups
- deleting groups
- duplicating groups
- reordering layers
- applying presets that create pageLook layers

Replace the current simple + menu with a searchable categorized library.

Tabs:

- Portrait
- Print
- Looks

Each item card should show:

====================================================================
PART 9 — Rendering rules
====================================================================

Canvas rendering:
- Normal layers render as they do today.
- ImageLayer renderer reads imageAdjustments and applies filters to that image only.
- PageLook layers render at the end, above normal layers.
- Legacy/classic adjustment layers do not affect live rendering when disabled.
- Group layers themselves do not render as canvas objects.
- Group visibility/opacity affects children through simple logic only, no cache.

Performance:
- Do not run heavy filters synchronously on every drag or zoom.
- During dragging/transforming, it is acceptable to temporarily render a simpler preview if needed.
- Full-quality rendering/export should apply the same adjustment parameters at final quality.
- If advanced filters are too heavy for live canvas, use preview quality or defer to export renderer.

Error handling:
- If an adjustment filter fails, render the original image and log a debug event.
- Never hide the image because a filter failed.
- Never throw inside render path for missing dependencies.
- Missing dependencies should produce Limited / Requires Dependency status.

Debug events:
- adjustment:classic-render-disabled
- adjustment:image-filter-failed
- pageLook:render-failed
- preset:missing-dependency
- group:normalized-invalid-parent
- group:removed-pageLook-from-group

====================================================================
PART 10 — Acceptance tests
====================================================================

Stability:
- Existing projects open and images remain visible.
- Adding any preset does not make images disappear.
- Adding a Page Look does not freeze the editor.
- Switching pages repeatedly does not show blank canvas.
- Dragging images remains responsive.

Image Adjustments:
- Select one image and apply brightness: only that image changes.
- Reset restores original appearance.
- Copy/paste adjustments works.
- Apply preset to 10 selected images is one undoable action.
- Applying to all images on page does not affect text, frames, or decorative elements.

Page Look:
- Page Look always renders above all normal layers.
- Page Look affects text/background/images because it is a page effect.
- Page Look can be hidden/deleted/duplicated.
- Page Look cannot be inserted into a group.
- Page Look opacity/strength works.

Presets:
- Each preset has a definition, category, description, tags, recommended apply mode.
- Missing optional dependencies show Limited but still allow fallback.
- Missing required dependencies disable apply or show a safe fallback.
- Recently Used works.
- Recommended tab changes based on selection.

Groups:
- Moving layers into groups does not make them disappear.
- Group visibility hides/shows children.
- Delete group only ungroups children.
- Delete group with children removes children.
- Normalizing groups prevents stale childIds/parentId bugs.

Export/Print:
- Image adjustments are included in export/print.
- Page Look layers are included in export/print.
- Preview quality does not limit final quality.
- Existing fill/crop behavior is unchanged.
- Existing photo print/collage/product modes continue to work.

====================================================================
FINAL IMPORTANT INSTRUCTION
====================================================================

Prioritize stability over Photoshop-like completeness.
This is not a request to build full Photoshop adjustment layers now.
The goal is to keep the full creative preset/library vision but implement it through:
- non-destructive ImageLayer adjustments
- Smart Presets
- lightweight Page Look layers
- safe group metadata

Classic adjustment layers that affect all layers below should remain disabled/experimental until a real CompositeRenderer is built later with OffscreenCanvas, render plans, partial bounds, and proper cache invalidation.
